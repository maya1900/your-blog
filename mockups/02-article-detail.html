TanStack Query 缓存策略:从 staleTime 到失效设计

顾远舟
2026/05/20 8 分钟阅读 1,247 阅读

默认的 staleTime: 0 不是 bug,是 feature。但 90% 的项目都不想要这个 feature。这篇文章捋清楚 stale、gc、refetch 三个时间窗口怎么配,缓存就稳了。

上周 review 一个组员的 PR,改的是个列表页接口请求。她把所有 query 都加了 refetchOnWindowFocus: false,理由是"切回标签页就刷数据太烦了"。我看完没说话,只问了一句:那你 staleTime 怎么设的?她说没设过,默认就好。

这是 TanStack Query 用户里最常见的误解之一 — 把"何时认为数据过期"(stale)和"何时重新拉数据"(refetch)混为一谈。实际上文档里写得明明白白,它们是两个独立的旋钮。

为什么需要 staleTime

Query 的生命周期里有三个状态:fresh、stale、inactive。staleTime 控制的是 fresh → stale 的过渡时间,gcTime(旧名 cacheTime)控制的是 inactive → garbage collected 的过渡时间。两个时间互不影响。

默认值的陷阱

默认 staleTime: 0 的设计哲学是"数据一拿到就过期",所以任何重新挂载、窗口聚焦、网络重连都会触发 refetch。这在数据高度动态的场景(交易盘口、协作编辑)是对的,但在博客文章详情、用户资料这种"几小时内根本不会变"的场景下,纯粹是浪费。 

const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      // 5 分钟内数据视为新鲜,不会触发 refetch
      staleTime: 5 * 60 * 1000,
      // 10 分钟没人订阅就 GC,腾出内存
      gcTime: 10 * 60 * 1000,
      refetchOnWindowFocus: true,
      retry: 2,
    },
  },
})

注意我没有关 refetchOnWindowFocus。原因是:有了 staleTime,窗口聚焦时 Query 会先看数据是否还 fresh,fresh 就什么都不做。所以关键不是关 refetch,而是定义"什么时候才算 stale"。

实战策略

下面是我在三个不同项目里沉淀出来的 staleTime 选型表:

  1. 静态资源类(分类列表、标签云、配置项):Infinity,手动失效
  2. 低频变动类(文章详情、用户资料):5 分钟,允许 refetchOnWindowFocus
  3. 实时性中等(评论列表、文章列表):30 秒,允许 refetchOnReconnect
  4. 实时性强(在线状态、协作光标):0,配合 refetchInterval

对应的失效触发,我习惯只用三种方式:

  • Mutation 成功后 invalidateQueries 精准命中相关 key
  • 路由切换时清理某个 namespace 的 queries
  • 用户主动刷新(下拉、按钮)时调 refetch
缓存最难的不是何时拉数据,而是何时不要拉数据。staleTime 是你给自己设的边界,refetchOnWindowFocus 是你给用户的恩典 — 不要把这两件事弄混。

这套策略落到代码上,关键不在 config 多花哨,而在 query key 怎么设计。后面会单独写一篇讲 key 的层级化命名 — 让 invalidateQueries 能像 rm -rf ./posts/* 一样精确。

那个组员后来把 PR 改了。我们一起把 defaultOptions 抽到了 queryClient.ts,各 page 不再传一堆零散参数。代码少了 60 行,bug 少了一类。
